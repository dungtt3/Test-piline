# ADR-011: Trivy DB đóng băng vào image, quét offline

- **Bối cảnh:** Trivy cần vulnerability DB (và Java DB) để quét SCA/secret/misconfig. Mặc định Trivy tải DB từ mạng mỗi lần chạy — không chấp nhận được cho môi trường lab/air-gapped và làm job phụ thuộc mạng, chậm, dễ flaky. Spec Phase 3 Phần 3 yêu cầu chiến lược offline có ADR.
- **Quyết định:** (1) `Dockerfile` chạy `trivy image --download-db-only` (+ java-db) **lúc build image**, nhúng DB vào layer. (2) `entrypoint.sh` set `TRIVY_SKIP_DB_UPDATE=true` (và `TRIVY_SKIP_JAVA_DB_UPDATE=true`) mặc định → runtime **không bao giờ** chạm mạng. (3) Chạy 2 lần trên cùng DB đã cache: lần 1 xuất SARIF vào `/results`, lần 2 xuất JSON gốc vào `/artifacts` (an toàn hơn `--output` kép giữa các version).
- **Hệ quả:** Job Trivy chạy hoàn toàn offline, ổn định. Đánh đổi: DB **đóng băng theo ngày build image** — CVE mới sau ngày đó không được phát hiện tới khi rebuild. Chấp nhận cho v1 lab; production nên có pipeline rebuild image adapter định kỳ (ghi `docs/backlog.md`). Nghiệm thu e2e thật (pull image ~ vài trăm MB + DB) hoãn như ADR-006 do giới hạn tài nguyên; adapter tuân thủ đầy đủ contract và verify tĩnh (cú pháp shell, ≤50 dòng, SARIF native của Trivy đã có SecurityEnricher xử lý ở M1).
